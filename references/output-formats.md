# 输出格式

当用户需要某种特定产物形状，而不是简单点评时，读取本文件。

## 0. 配套阅读

- 通用规则与证据分层：看 [core-guidelines.md](./core-guidelines.md)
- JD 与公司映射：看 [jd-and-company-mapping.md](./jd-and-company-mapping.md)
- 长报告与专项文档模板：看 [detailed-report-template.md](./detailed-report-template.md)
- 风险与自检：看 [risk-and-self-check.md](./risk-and-self-check.md)

## 1. 先确定唯一主受众

不要把多种 persona 混在一组 bullet 里。

常见受众：

- 实习 / 校招简历
- 资深工程师 / 架构师简历
- 面试准备
- JD 匹配分析
- 面向具体公司的重排版本
- 项目接手说明

默认优先输出实习 / 校招版本；只有用户明确要求时，才展开资深工程师 / 架构师版本。

如果用户同时要多个受众版本，复用同一套事实，分章节输出。

## 2. 推荐的精简输出

### 快速点评版

适用于用户问“这个项目怎么样”“有哪些亮点”。

结构：

1. 项目画像
2. 核心亮点
3. 可写边界
4. 最值得补的缺口

### 简历版

适用于用户问“简历怎么写”“哪些能写进项目经历”。

每条 bullet 尽量包含：

- 业务问题
- 技术方案
- 架构或工程价值

没有代码证据时不要写指标。

### 面试版

适用于用户问“面试怎么讲”“哪些点能展开”。

每个点尽量包含：

- 做了什么
- 为什么这么设计
- trade-off
- 面试官大概率怎么追问
- 哪个地方最容易吹过头

### JD 映射版

适用于用户问“和某岗位匹不匹配”。

重点输出：

- 匹配能力项
- 代码证据
- 安全表述
- 明显缺口

## 3. 推荐的长报告结构

只有用户明确要详细报告时再展开。

### 项目技术画像

总结：

- 系统类型
- 主链路
- 核心对象
- 当前成熟度
- 最强匹配岗位
- 最弱匹配岗位

### 亮点排序

对每个亮点给出：

- 证据等级：`S/A/B`
- 资深度层级：`L1/L2/L3/L4`
- 分类
- 代码证据
- 安全的简历写法
- 面试风险边界

### 业务背景重建

拆成三段：

- 代码明确可见的事实
- 可强推导的诉求
- 仍然不确定的地方

### 风险审查

列出：

- 高风险简历表述
- 高概率面试陷阱
- 没证据时绝对不要写的点

### 改进路线图

按下面几个维度排序：

- 简历价值
- 工程价值
- 实现成本

## 4. 公司与岗位映射

只有用户明确要公司口径时再做。

### AI 应用 / Agent / RAG 岗位

重点强调：

- 应用架构
- tool 编排
- 检索链路
- prompt 与 workflow 管理
- 可观测性和失败处理

不要把应用层优化写成模型侧优化。

### 平台工程岗位

重点强调：

- 抽象边界
- 可复用模块
- 配置化、治理、扩展性
- 稳定接口

### 分布式 / 存储岗位

只有代码确实体现这些内容时才强调：

- 数据分布
- 元数据
- 一致性
- 恢复能力
- 检索基础设施

否则保守停留在平台 / 数据访问口径。

### Java 后端岗位

重点强调：

- 模块设计
- 事务与数据访问
- 中间件使用深度
- 可观测性与工程质量
- 并发或 JVM 证据（仅在真实存在时）

## 5. 1 分钟和 5 分钟项目讲法

### 1 分钟版本

只保留：

- 这是什么系统
- 主链路做什么
- 你做了什么或优化了什么
- 为什么重要

### 5 分钟版本

覆盖：

1. 项目背景
2. 系统类型与主链路
3. 你负责的模块
4. 2-3 个难点
5. 关键 trade-off
6. 稳定性或治理工作
7. 岗位匹配点和剩余缺口

## 6. 常用 trade-off 提示

可以用这些角度把亮点讲深：

- 简单实现 vs 可扩展性
- 一致性 vs 吞吐
- 同步处理 vs 异步处理
- 检索质量 vs 延迟
- 模型效果 vs token 成本
- 局部优化 vs 系统复杂度

## 7. 安全措辞示例

优先：

- `参与核心链路设计与实现`
- `负责模块抽象与落地`
- `围绕检索/工作流链路做工程化改造`
- `基于代码事实可确认`

避免：

- `主导整体架构`
- `从 0 到 1 搭建平台`
- `行业领先`
- `高并发架构负责人`
