# Java 后端评审口径

当仓库以 Java 为主，或者用户目标是 Java 后端岗位时，读取本文件。

## 0. 配套阅读

- 通用规则与证据边界：看 [core-guidelines.md](./core-guidelines.md)
- Java 后端 JD 与公司口径：看 [jd-and-company-mapping.md](./jd-and-company-mapping.md)
- 长报告模板：看 [detailed-report-template.md](./detailed-report-template.md)

## 1. 用保守口径评估 Java 深度

不要把框架标准用法直接当成后端深度。

建议把观察结果分成三层：

- 框架标准实践
- 扎实的工程实现
- 有架构价值的设计与治理

## 2. 哪些证据更能说明深度

### Java 核心与并发

优先寻找：

- 自定义 `ThreadPoolExecutor`
- `CompletableFuture` 异步编排
- 有明确取舍依据的锁使用
- 原子类或并发容器
- 上下文透传与清理

如果这些都没有，就不要硬写并发深度。

### JVM 与性能

优先寻找：

- JVM 启动参数
- GC 日志
- 内存诊断信息
- 堆或线程排障痕迹
- 明确的性能脚本或诊断文档

没有这些证据时，更适合写 `理解 JVM 内存模型和 GC 机制`，不要写 `完成 JVM 调优`。

### Spring 生态深度

优先寻找：

- 明确的事务边界设计
- 基于 AOP 的统一能力封装
- 事件驱动解耦
- 自定义校验
- Actuator 扩展
- 有意义的测试切片

如果项目基本是 `@RestController + @Service + @Repository`，更适合表述为常规 Spring Boot 业务开发。

### 数据库与数据访问

优先寻找：

- 有明确取舍的表结构设计
- 索引设计
- 批量操作
- 分页策略
- SQL 调优证据
- 数据访问层抽象质量

不要把基础 CRUD 写成“数据库设计能力突出”。

### 中间件与治理

优先寻找：

- MQ 可靠性或幂等设计
- Redis 策略与一致性处理
- 调度任务的重试或分片处理
- 分布式锁或去重保护

如果只是接入了依赖，但没有治理设计，只能写“完成接入”，别写“主导治理”。

### API 与工程质量

优先寻找：

- 统一响应与异常处理
- 参数校验
- 请求日志
- 审计能力
- 分层边界
- 测试支撑

这些往往决定一个项目是普通 CRUD，还是更像有质量意识的后端工程。

## 3. 安全的 Java 后端表述

优先：

- `基于 Spring Boot 完成核心业务模块开发`
- `围绕数据访问与接口链路完成抽象与优化`
- `参与缓存/MQ/任务调度接入与治理`
- `对异常处理、日志与分层结构进行规范化改造`

避免：

- `精通 JVM 调优`
- `主导微服务架构设计`
- `完成全链路 SQL 调优`
- `主导中间件治理体系`

只有在代码证据非常扎实时，才使用更强的说法。
