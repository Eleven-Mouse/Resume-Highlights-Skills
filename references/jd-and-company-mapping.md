## 2. AI 推理 / 模型 / 通信 / 计算性能优化 JD

只有代码真实出现下列证据，才允许往该方向表达：

### 模型优化

- 量化：INT4 / INT8 / FP8
- 稀疏化 / Sparse Attention
- KV Cache / Prefix Cache
- Speculative Decoding
- MoE 路由
- 模型性能 Benchmark

### 通信优化

- RDMA
- TCP 协议栈优化
- 分布式训练 / 推理通信
- NCCL / HCCL / Gloo
- 通信拓扑 / 流量调度

### 计算优化

- CUDA / ROCm / Triton / MLIR / TVM
- Transformer 核心算子
- GEMM / 算子融合 / 向量化执行
- GPU Profiling / Nsight / PyTorch Profiler

### 全链路协同优化

- batching / continuous batching
- prefix cache / KV cache
- speculative decoding
- 计算-通信 overlap
- 自动化压测与调优 pipeline

如果只是调 OpenAI / Claude / Gemini / 本地模型 API：

- 可以写 `模型调用治理 / LLM 应用链路优化`
- 不能写 `主导模型推理优化`

## 3. 分布式系统 / 存储 / 平台架构 JD

重点检查是否涉及：

- 对象存储 / 文件存储 / 向量存储 / 搜索引擎
- 元数据管理
- 数据分片 / 副本 / 一致性 / 恢复
- 读写路径优化 / Cache 层 / 索引结构
- 大规模数据处理
- 多租户隔离 / 数据权限 / 审计 / 生命周期管理

如果项目不是存储系统，不要硬包装成存储架构。更稳妥的说法：

- 涉及数据存储与检索链路设计
- 参与核心数据读写链路优化
- 围绕数据访问、缓存和检索链路做工程治理
- 具备向分布式存储 / 数据平台方向迁移的项目基础

## 5. 工程深度分层

对每个亮点同时判断两个维度：

### 证据强度

- `S`：代码直接证明，强烈建议写简历
- `A`：代码较强支撑，可写但要克制
- `B`：更适合面试延展，不建议直接写成已落地成果

### 工程深度层级

- `L1`：基础工程卫生项
- `L2`：高级工程师项
- `L3`：复杂系统设计与治理项
- `L4`：平台化 / 标准化 / 方向性建设项

`L4` 必须慎给，除非代码和项目事实非常扎实。

判断时重点看：

- 是否解决了复杂系统问题，而不是只完成业务功能
- 是否体现清晰 trade-off，而不是堆技术名词
- 是否体现跨模块、跨链路、跨团队的设计能力
- 是否具备稳定性、扩展性、治理性、演进性
- 是否能在三层追问后讲住

## 7. 风险边界

### AI 性能优化方向

如果没有量化、通信优化、算子优化、GPU / AI 芯片、Benchmark / Profiler 证据，必须明确写：

`当前项目不建议主打 AI 性能优化方向岗位。`

### 分布式 / 存储方向

如果没有元数据、副本、一致性、数据分片、恢复等证据，必须明确写：

`不建议硬写大规模分布式存储架构经验；更适合保守表达为数据访问、检索、缓存或平台化工程能力。`
