## 1. JD 差距分析模板

建议至少覆盖：

### AI 应用开发工程师 JD 匹配度

- 已匹配能力
- 部分匹配能力
- 未发现证据的能力
- 简历主打建议
- 面试风险
- 补齐建议

### Agent / RAG / Workflow JD 匹配度

- 已匹配能力
- 部分匹配能力
- 未发现证据的能力
- 简历主打建议
- 面试风险
- 补齐建议

### AI 性能优化方向 JD 匹配度

- 已匹配能力
- 部分匹配能力
- 未发现证据的能力
- 是否建议主打该 JD
- 如果不建议，原因是什么
- 如果想补齐，需要补哪些工程能力

如果没有模型量化、通信优化、算子优化、GPU / AI 芯片、Benchmark / Profiler 证据，必须明确写：

`当前项目不建议主打 AI 性能优化方向岗位。`

### 分布式 / 存储方向 JD 匹配度

- 已匹配能力
- 部分匹配能力
- 未发现证据的能力
- 是否建议主打该 JD
- 如果不建议，原因是什么
- 如果可迁移，应该怎么表达

### Java 后端工程师 JD 匹配度

- Java 核心技术匹配度
  - 并发编程能力评估
  - JVM 理解程度
  - I/O 模型使用
- Spring 生态匹配度
- 数据库能力匹配度
- 中间件治理匹配度
- 工程规范匹配度
- 微服务能力匹配度（如适用）
- 已匹配能力
- 部分匹配能力
- 未发现证据的能力
- 最容易被深挖的 3 个点
- 最需要补齐的 3 个能力
- 简历主打建议
- 面试风险

## 3. 最终自检清单

输出前逐条检查：

1. 这条亮点是否能被代码事实直接支撑？
2. 如果面试官问“代码在哪里”，是否能回答？
3. 如果面试官追问“为什么这么设计”，是否能讲清 trade-off？
4. 如果面试官追问“有没有指标”，是否存在真实证据？
5. 如果没有指标，是否避免写百分比和规模数据？
6. 是否错误使用“主导”“整体架构”“全链路”“高并发”“千万级”等高风险词？
7. 是否错误使用“主导模型推理优化”“量化优化”“RDMA”“CUDA 算子优化”等无证据词？
8. 是否把建议补齐的优化点写成了已实现？
9. 是否符合目标 seniority？
10. 是否给出了每个亮点第三层追问的薄弱环节？
11. 是否把 S 级亮点整理成了可复述、可面试、可写简历的专项文档？
12. S 级亮点文档是否能让刚接手项目的人看懂？
13. 是否解释了核心术语？
14. 是否给出了代码阅读顺序？
15. 是否说明了入口、主链路、数据流和异常场景？
16. 是否明确标注了哪些是已实现，哪些只是建议补齐？
17. 是否对 AI 应用开发工程师 JD 做了匹配分析？
18. 是否对 AI 性能优化方向 JD 做了风险隔离？
19. 是否对分布式 / 存储方向 JD 做了保守判断？
20. 是否避免把普通 LLM API 调用包装成模型底层优化？
21. 是否对 Java 后端工程师 JD 做了匹配分析？
22. 是否区分了“框架标准用法”和“有技术深度的设计与优化”？
23. 是否避免把普通 Spring Boot CRUD 包装成架构能力？
24. 是否避免虚构 JVM 调优、并发优化、SQL 调优等无证据经历？
25. 是否避免把使用 `@Transactional` 等同于“深入理解事务管理”？
26. 是否避免把使用 Redis 缓存等同于“主导缓存架构设计”？

## 4. 特别注意

- 先扫描、再判断、再排序，不要一上来只盯着 controller / service
- 对每个亮点都要问：这是普通实现，还是值得深挖的点
- 对每个简历表述都要问：三层追问后还能讲住吗
- 如果没有证据支撑，请直接说“未发现足够证据”
- 如果项目更偏普通业务系统，要如实输出，不要强行包装成复杂分布式系统
- 如果项目只是 LLM API 调用，不要包装成模型推理优化
- 如果没有量化、算子、通信、GPU、Benchmark、Profiler，不要写 AI 性能优化方向经历
- 如果没有分布式存储、元数据、副本、一致性、数据分片等证据，不要写大规模分布式存储架构经验
- 不要为了显得高级而硬套 TCC、Saga、Raft、多活、CQRS、Event Sourcing、RDMA、CUDA、Triton、MoE、Speculative Decoding 等概念
- 如果项目是 Java 技术栈，不要把 `@Transactional`、`@Async`、`@Cacheable` 等注解直接等同于架构深度
- 不要把使用线程池等同于“并发编程能力突出”，要看是否有自定义参数、拒绝策略、监控等证据
- 不要把引入 Redis 缓存等同于“缓存架构设计”，要看是否有一致性、防护策略、数据结构选型等治理证据
- 不要把使用 MQ 等同于“消息中间件架构设计”，要看是否有可靠性、幂等、死信、顺序等治理证据
- 不要把使用 Spring Boot 开发业务等同于“Spring 生态深度使用”，要看是否有 AOP 统一能力、事件解耦、自定义扩展等证据
- 不要把简单索引创建等同于“数据库优化能力突出”，要看是否有慢 SQL、EXPLAIN、N+1 消除、分页优化等系统性证据
- 所有输出都要克制、专业、可验证
