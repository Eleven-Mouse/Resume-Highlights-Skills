- 串行链路改并行
- 网络 IO 优化
- 请求折叠 / 合并请求
- 核心链路缓存
- 批量 SQL / 批处理
- 连接池 / 线程池治理
- 异步化 / MQ 解耦
- 超时 / 重试 / 熔断 / 降级
- 慢查询治理
- 数据结构与序列化优化
- 大对象 / 大响应裁剪
- 热点 Key / 热点数据保护
- RAG 检索链路优化
- LLM 调用成本与延迟优化
- Agent / Workflow 执行优化
- Tool 调用安全与性能治理
- 存储 / 检索读写路径优化
- 模型推理链路优化（仅在有证据时）

必须严格区分：

- `【当前已实现】`：代码已有明确证据，可写简历
- `【强推导可写】`：代码有部分证据，措辞必须克制
- `【建议补齐】`：当前未实现，只能作为优化方案

## 3. 单个优化点模板

### 优化点：{标题}

#### 优化类型

- 性能优化 / 稳定性优化 / 成本优化 / 可用性优化 / 工程治理优化 / AI 应用链路优化 / 存储检索优化 / 推理优化
- 适配岗位：AI 应用开发工程师 / 平台工程师 / Java 后端工程师 / 分布式系统工程方向 / 存储检索工程方向 / AI 性能优化工程方向
- 当前状态：`【当前已实现】/【强推导可写】/【建议补齐】`
- 简历推荐级别：`S / A / B`

#### 原始问题

说明优化前可能存在的问题，例如：

- 串行等待
- 重复请求
- 高频 DB 查询
- 外部服务 IO 阻塞
- 慢查询 / 大分页 / N+1
- LLM / embedding / rerank 调用成本高
- Tool 调用不受控
- Workflow 某一步失败导致整体失败
- RAG 召回慢或结果不稳定
- 存储读写路径过长
- 推理服务吞吐瓶颈

如果代码没有明确证据，只能写“从代码结构看存在优化空间”。

#### 当前代码事实

- 文件路径
- 类 / 方法
- 调用链
- 数据库 / Redis / MQ / 外部服务
- LLM / VectorDB / Tool / Workflow
- 关键代码摘要

#### 优化方案

要写具体做法，不要只写概念。

##### 示例：串行改并行

- 将多个互不依赖的外部调用改为并行编排
- 设置统一超时、失败兜底和结果聚合
- 对强依赖结果保留 fail-fast，对弱依赖结果允许降级

##### 示例：请求折叠

- 对同一资源 ID / query / cache key 的短时间重复请求做 singleflight 合并
- 首个请求负责加载数据，后续请求等待同一个 future / promise
- 避免热点场景下重复打 DB / 外部 API / LLM

##### 示例：核心链路缓存

- 对读多写少、变更频率低的数据引入 Redis / 本地缓存
- 设置 TTL、主动失效、版本号或双删策略
- 对热点数据增加本地缓存或预热机制

##### 示例：RAG 链路优化

- 对 query rewrite、召回结果、rerank 结果做缓存
- 控制 chunk 数量和上下文长度
- 对空召回、低置信召回做兜底
- 增加检索日志和 Eval 闭环

##### 示例：Workflow 执行优化

- 将无依赖节点并行化
- 为每个节点设置超时、重试和降级
- 保存节点执行状态，支持失败恢复
- 记录节点输入输出，便于排查

##### 示例：AI 推理优化

仅在代码有证据时使用：

- batching / continuous batching
- KV Cache / Prefix Cache
- speculative decoding
- 量化 / 稀疏化
- benchmark / profiler 体系

#### Trade-off

至少解释代价：

- 并行化带来的线程池压力 / 上下文传播问题
- 缓存带来的一致性问题
- 请求折叠带来的等待放大 / 单点慢请求问题
- 异步化带来的最终一致性和补偿复杂度
- 批处理带来的实时性下降
- 降级带来的结果完整性下降
- RAG 召回更全带来的延迟上升
- 模型成本下降可能带来的效果下降

#### 可写简历表述

##### 如果当前已实现

可以写成：

`围绕核心链路性能瓶颈，完成 xxx 优化，通过 xxx 机制降低重复 IO / 减少串行等待 / 提升链路稳定性。`

##### 如果只是建议补齐

只能写成：

`针对 xxx 链路设计了缓存 / 请求折叠 / 并行编排优化方案，作为后续性能治理方向。`

#### 高频追问

1. 为什么这个地方能并行？依赖关系怎么判断？
2. 缓存一致性怎么保证？
3. 请求折叠如果首个请求很慢怎么办？
4. 线程池参数怎么设置？
5. 超时、重试、降级怎么配合？
6. 有没有压测或指标证明？
7. RAG 效果怎么评估？
8. Agent / Tool 调用失败怎么兜底？
9. Workflow 节点失败怎么恢复？
10. 如果说 AI 性能优化，具体优化的是模型层、通信层还是计算层？

#### 能写 / 不要写

能写：

- 优化核心链路
- 降低重复请求
- 减少不必要 IO
- 引入缓存降低数据库访问压力
- 对独立调用进行并行编排
- 增加超时和降级保护
- 完善 RAG 链路观测
- 优化 LLM 调用成本与稳定性
- 增强 Workflow 执行可靠性

不要写：

- QPS 提升 xx%
- 延迟降低 xx%
- 支撑千万级并发
- 彻底解决性能问题
- 零故障
- 全链路性能治理
- 主导模型推理优化
- 主导 INT4 / INT8 / FP8 量化
- 主导 RDMA 通信优化
- 主导 CUDA / Triton 算子优化

## 4. 最值得补充的方向

当代码中暂时没有足够证据时，优先考虑以下补齐项：

1. RAG Eval 评测体系
2. Prompt 模板版本管理
3. Tool 调用权限与审计
4. Workflow 节点级状态与恢复
5. LLM 调用超时 / 重试 / 降级
6. 检索链路缓存与请求折叠
7. 文档解析 / embedding / 建索引异步化
8. 核心链路可观测性
9. 多租户 / 数据权限隔离
10. Benchmark / 压测体系
11. 存储 / 检索链路优化
12. AI 推理优化能力建设（仅在项目方向适合时）

每条建议都要写清楚：

- 建议补在哪个模块
- 当前代码已有基础
- 当前缺口
- 为什么值得补
- 补完后能形成什么亮点
- 更适合哪类岗位 / 哪个 JD
- 风险和代价
- 为什么现在不能写成已实现
