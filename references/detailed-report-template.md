# 详细报告模板

本文件承接原大纲中“长报告输出结构”“S 级亮点专项文档”“1 分钟 / 5 分钟讲法”“最终结论”相关模板。

## 1. 何时使用

只有用户明确要详细报告、专项文档或面试长稿时，再读本文件。

## 2. 详细报告总结构

推荐按以下顺序输出：

1. 项目技术画像
2. 业务背景重建
3. 系统全景扫描
4. TOP 项目亮点排序
5. 最适合写进简历的亮点重写
6. 最适合面试展开的亮点
7. 按目标公司和 JD 重排
8. 定向筛选后的核心亮点
9. S 级亮点专项文档
10. 系统优化专项文档
11. JD 差距分析
12. 简历风险审查
13. 1 分钟项目介绍
14. 5 分钟深挖版本
15. 最终结论

## 3. 项目技术画像

建议用 300~500 字总结：

- 系统类型
- 主链路
- 核心业务对象
- 更偏一致性还是吞吐 / 可用性
- 当前成熟度
- 最适合突出哪些岗位能力
- 与 AI 应用、AI 性能、分布式 / 存储、Java 后端 JD 的匹配度
- 最大价值与最大短板

要求：

- 只基于代码事实
- 不确定就直接写“不确定”
- 不要把普通 LLM API 调用包装成模型推理优化

## 4. 业务背景重建模板

### 代码已明确体现的业务事实

每条建议包含：

- 业务事实
- 证据
  - 文件路径
  - 类 / 方法
  - 表 / 接口 / 配置
- 可写入简历程度
- 适配 JD
- 风险边界

### 可强推导的业务诉求

每条建议包含：

- 推导诉求
- 推导依据
- 可写边界
- 适配 JD
- 不可夸大点

### 仍不确定的业务背景

每条建议包含：

- 不确定点
- 当前缺少的证据
- 对应 JD 风险
- 面试中应如何保守表达

## 5. 系统全景扫描模板

建议输出“架构全景图的文字版”，至少覆盖：

- 模块划分
- 核心链路
- 核心表
- 缓存
- MQ
- 定时任务
- 外部依赖
- 稳定性与治理设施
- 安全与权限设施
- AI / RAG / Agent / Workflow / DSL 能力
- 模型调用 / 推理 / Eval / Benchmark 能力
- 存储 / 检索 / 元数据能力
- 工程质量设施

推荐格式：

`用户请求 / 外部任务 -> Controller / Handler -> Application Service -> Domain Service / Strategy / Workflow / Agent -> Repository / DAO / Client / Tool / Retriever -> DB / Redis / MQ / VectorDB / LLM / Object Storage / 外部 API -> 日志 / Trace / Metrics / Eval / 异常处理`

## 6. 亮点排序模板

每个亮点建议包含：

1. 标题
2. 亮点级别：`S/A/B`
3. 资深度层级：`L1/L2/L3/L4`
4. 亮点分类
5. 结论归类：`【代码已实现】/【强推导亮点】/【可扩展设计】`
6. 主导度判断
7. 一句话简历表述
8. 技术拆解
   - 背景问题
   - 方案设计
   - 实现机制
   - trade-off
   - 收益与价值
9. 代码证据
10. 为什么这是亮点而不是普通实现
11. JD 适配映射
12. 面试官追问点
13. 风险提示
14. 反吹牛审查
15. 能写 / 不要写

## 7. 简历重写模板

常见输出版本：

1. Java 后端工程师版本
2. AI 应用开发工程师版本
3. 前端开发工程师版本
4. 全栈工程师版本
5. 主项目版本
6. 次项目版本

要求：

- 专业、克制、可信
- 不虚构数据
- 最好体现“业务问题 + 技术方案 + 工程价值”
- 版本之间复用同一套事实，只调整重心
- 仅当仓库存在真实前端代码、页面交互、状态管理或前端构建配置等证据时，才输出前端 / 全栈版本

## 8. 面试展开模板

每条亮点尽量包含：

- 面试官可能怎么问
- 他真正想考什么
- 怎么答更像目标岗位工程师
- 哪些答案会显得浅
- 最佳深度边界
- 如果被追问代码位置，应该指向哪里
- 如果被追问指标，在没有数据时如何保守回答

## 9. 公司与 JD 重排模板

常见重排目标：

- 字节 AI / RAG / Agent / 数据智能
- 阿里增长 / 平台 / 核心链路
- 腾讯稳定性 / 平台治理
- Google SWE / L4-L5
- Amazon SDE / SDE II
- Binance Backend / AI Backend
- AI 应用开发工程师 JD
- AI 性能优化方向 JD
- 分布式 / 存储方向 JD
- Java 后端工程师 JD

每条说明建议包含：

- 为什么适合
- 应该怎么写
- 不应该怎么写
- 对应代码证据
- 风险边界

## 10. S 级亮点专项文档模板

S 级亮点必须做到“刚接手项目的小白也能看懂”。

建议结构：

1. 这个亮点一句话说人话
2. 业务背景，小白版
3. 核心术语解释
4. 代码入口在哪里
5. 核心代码文件清单
6. 主链路逐步解释
7. 数据是怎么流动的
8. 为什么这样设计
9. Trade-off
10. 异常场景和兜底
11. 为什么值得写进简历
12. 简历表述
13. 面试三层追问准备
14. 面试官验证问题
15. 第三层最容易暴露的薄弱环节
16. 新人接手阅读路线
17. 风险边界

术语解释建议用表格：

| 术语 | 小白解释 | 在代码中的体现 |
|---|---|---|
| 幂等 | 多次请求只生效一次 | xxx |
| RAG | 先检索再交给模型回答 | xxx |
| Agent | 能决定下一步调用哪个工具或流程的程序 | xxx |
| Workflow | 多个步骤按顺序或条件执行 | xxx |

## 11. 1 分钟项目介绍模板

要求：

- 不超过 250 字
- 适合面试开场白
- 只讲最核心业务、主链路、本人贡献、技术价值
- 不堆名词
- 不虚构规模和数据

常见版本：

1. 克制专业版
2. 更偏工程实现版
3. 更偏业务结果版
4. AI 应用开发工程师版
5. 平台 / 全栈工程师版

如果 AI 性能优化证据不足，不要硬出该版本。

## 12. 5 分钟深挖模板

建议顺序：

1. 项目背景
2. 系统定位
3. 核心链路
4. 我参与 / 负责的模块
5. 最核心的 2~3 个技术难点
6. 关键设计取舍
7. AI 应用 / RAG / Agent / Workflow 能力，如有
8. 系统优化与治理
9. 风险边界和后续演进
10. 与目标 JD 的匹配点

要求：

- 像自然讲项目，不像背稿
- 每个技术点都能接住追问
- 对不确定的地方保守表达

## 13. 最终结论模板

建议给出：

1. 最值得打的 3 张牌
2. 最不该硬吹的 3 个点
3. 如果定位为主项目，怎么包装
4. 如果目标是 AI 应用开发工程师，怎么包装
5. 如果目标是 Agent / RAG / Workflow 相关工程岗位，怎么包装
6. 如果目标是前端 / 全栈工程师，怎么包装
7. 如果目标是分布式 / 存储方向岗位，是否适合、如何保守表达
8. 如果目标是 AI 性能优化方向岗位，是否适合、为什么
9. 如果目标是 Java 后端工程师，怎么包装
10. 如果只能选 3 条写进简历，推荐哪 3 条
11. 如果只能选 1 条作为主讲点，推荐哪 1 条
12. 如果需要补代码，最优先补哪 3 个优化点
13. 推荐投递岗位方向排序
